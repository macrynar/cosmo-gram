## TL;DR (60 sekund)

**Co budujemy:** Aplikacja webowa (PWA) łącząca astrologię + AI. Generuje natal chart, daje codzienne odczyty (transit + natal), pozwala porównać kompatybilność dwóch osób (synastria). Pozycjonowanie: **symboliczne lustro, nie wyrocznia** - ironic-distant w stylu Co-Star, ale po polsku i z autentycznym głosem (od koleżanki).

**Rekomendowany zakres pierwszej działającej wersji (7 dni intensywnej pracy, vibe coding):**
1. Natal chart + AI interpretacja (już masz lokalnie - dopracować prompt z koleżanką)
2. Daily reading (transit + natal w jednym prompcie, cache 24h)
3. Astro-Match (synastria) - feature kupowana, główne źródło przychodu
4. **Cosmogram Chat** - konwersacja z AI z kontekstem natalnego kosmogramu + dzisiejszych tranzytów (sprawdzone w Alcheme, główny differentiator)
5. Onboarding z fallbackiem "nie znam godziny urodzenia"
6. Auth + Stripe subscription + 7-dniowy trial
7. Landing z waitlistą przed launchem

**Rekomendacja monetyzacji:** Pure subscription, jeden tier. 29-39 zł/mc. Bez one-time micropayments. Deep dive PDFs i karty tarot - dopiero Phase 2.

**Co WYJEBANE z pierwszej wersji (do późniejszych iteracji):** tarot (zabity - konflikt pozycjonowania), kalendarz/moodboard (drogie w UX, robimy po pierwszych 100 płatnych), fazy księżyca jako osobna sekcja (mały dodatek, dorzucamy do dashboardu po Dniu 7), deep dive PDFs (Złoty Kompas, Bagaż Karmiczny - dopiero gdy są pieniądze z subskrypcji).

**Co weryfikujemy w weekend:** styl interpretacji koleżanki, system domów, jej rola/zaangażowanie, voice/tone, czy daje 10 sample interpretations jako ground truth.

## 1. Wizja produktu

### Pozycjonowanie (z brief.md, do potwierdzenia W0)
**Symboliczne lustro, nie wyrocznia.** Astrologia jako narzędzie introspekcji i refleksji, nie predykcji. Ironic-distant - może pasować do PL kultury, może nie. Test w W2-W3 IG/TikTok content.

### Differentiator (vs konkurencja)
| Konkurent | Co robią | Czego nie robią |
|---|---|---|
| Co-Star (US) | Hyper-personalised push, ironic copy | Brak głosu od żywego eksperta. PL = 0. |
| The Pattern | Psychologized, mood-based | Brak natal chart deep dive. Zero PL. |
| Astrolada (PL) | Tradycyjna, blog/YT | Brak personalizacji AI. Brak monetyzacji per user. |
| Sanctuary (US) | Tarot + astrology + live chat | Drogie ($14.99/mc), live readers, brak PL. |

**Nasz wedge:** PL-first + AI personalization + ekspert-partner (autentyczność) + ironic distance (nie wymaga wiary).

### [W0-DEPENDENT] Głos i ton
Trzy warianty do dyskusji z koleżanką:
- **Wariant A (Co-Star style):** "Mars w 10 domu właśnie pisze Twojego maila do szefa za Ciebie. Nie pomaga."
- **Wariant B (psychologized):** "Dzisiejszy tranzyt Marsa nad Twoim domem kariery generuje napięcie. To dobry moment by nazwać frustrację, nie reagować na nią."
- **Wariant C (klasyczny astrologiczny):** "Tranzyt Marsa w aspekcie kwadratury do Słońca w 10 domu - okres wzmożonych konfliktów zawodowych. Zaleca się rozwagę w decyzjach."

Rekomendacja: B z domieszką A (psycho-mainstream + lekki dystans). Ale **decyduje koleżanka** - to jej głos brzmi z appki, nie nasz.

## 4. Core user flows

### F1: Onboarding (z godziną urodzenia) - happy path
1. Landing → CTA "Sprawdź swój kosmogram (darmowo)"
2. Krok 1/3: Data urodzenia (date picker)
3. Krok 2/3: Miejsce urodzenia (autocomplete z geocoding API)
4. Krok 3/3: Godzina urodzenia + "Nie znam → pomiń"
5. Email/Google signup (gate dla zapisu wyniku)
6. Generuje natal chart + AI interpretacja (loading state z ciekawostkami, ~10-15s)
7. Wynik: 1 strona z chartem (wizualizacja) + 3-4 paragrafy interpretacji (Sun/Moon/Rising + 1 kluczowy aspekt)
8. CTA na dole: "Odblokuj pełną interpretację - trial 7 dni"

### F2: Onboarding (bez godziny urodzenia) - fallback
Różnice vs F1:
- W kroku 3: user wybiera "Nie znam godziny urodzenia"
- System ustawia czas = 12:00 (solar chart standard)
- Generuje **uproszczony chart** - bez znaku wschodzącego, bez domów, tylko planety w znakach
- Interpretacja focusuje się na: Sun, Moon (przybliżony), planety w znakach
- Komunikat: "Twój odczyt jest uproszczony bo nie znamy godziny. Dla pełnej analizy (znak wschodzący, domy) sprawdź akt urodzenia. Możesz uzupełnić później w ustawieniach."

### F3: Daily reading (paid feature, główny retention loop)
1. Push notification rano (8:00, customizable): "Twój dzisiejszy odczyt jest gotowy"
2. User otwiera apkę → automatyczne wygenerowanie / cache (1x na dzień)
3. Czytelny format: 1 nagłówek ("Dzień napięcia ale produktywny"), 2-3 paragrafy, 1 "co dziś zrobić / czego unikać"
4. CTA: "Zobacz Astro-Match" / "Twój miesiąc"

### F4: Astro-Match (synastria) - revenue driver
1. Już zarejestrowany user → klik "Astro-Match"
2. Form: imię + data + miejsce + godzina partnera (godzina opcjonalna z osobnym fallback)
3. Generuje:
   - Overall compatibility score (1-100, gamified)
   - 4 kategorie: Komunikacja, Namiętność, Wspólne wartości, Wyzwania
   - Każda kategoria: 2-3 zdania interpretacji + 1 actionable insight
4. Pierwszy match w trialu darmowy, kolejne = paid feature
5. Save - history matchów (cap np. 10 dla unsub, unlimited dla paid)

### F5: Subscribe / upgrade
- Trigger pointów: po pierwszym darmowym Astro-Match, po 3. daily reading, po 7-day mark
- Modal: 1 tier, 29-39 zł/mc (do A/B test), trial 7 dni
- Stripe Checkout, paywall przed dostępem do daily/match po trialu

## 5. Data model (high-level)

```sql
-- users
id (uuid, pk)
email (unique)
created_at, updated_at
auth_provider ('email' | 'google')

-- birth_data
id (uuid, pk)
user_id (fk → users.id, ON DELETE CASCADE)
relation ('self' | 'partner' | 'family' | 'other')
display_name (varchar) -- "Ja", "Partner", "Mama"
birth_date (date, required)
birth_time (time, nullable) -- NULL = unknown
birth_time_unknown (boolean, default false)
birth_place_name (varchar) -- "Warszawa, Polska"
birth_place_lat (decimal)
birth_place_lon (decimal)
birth_place_tz (varchar) -- "Europe/Warsaw"
created_at

-- charts (cache compute)
id (uuid, pk)
birth_data_id (fk)
chart_data (jsonb) -- pozycje planet, domów, aspektów
computed_at
ephemeris_version (varchar) -- żeby re-compute przy zmianie biblioteki

-- readings
id (uuid, pk)
user_id (fk)
reading_type ('natal' | 'daily' | 'synastry' | 'deep_dive_career')
chart_id (fk, nullable for daily)
synastry_partner_id (fk → birth_data, nullable)
input_context (jsonb) -- tranzyty dnia jeśli daily
ai_output_text (text)
ai_model (varchar) -- "claude-sonnet-4-6"
ai_prompt_version (varchar) -- "natal-v1.2"
token_count_in, token_count_out (int)
cost_usd (decimal)
generated_at

-- subscriptions
id (uuid, pk)
user_id (fk)
stripe_customer_id, stripe_subscription_id
status ('trialing' | 'active' | 'past_due' | 'canceled')
plan ('plus_monthly' | 'plus_annual')
trial_start, trial_end
current_period_end
canceled_at

-- analytics_events (lekka warstwa, complement do PostHog)
id, user_id, event_name, properties (jsonb), occurred_at
```

**Decyzje projektowe:**
- `birth_data` osobne od `users` bo user może mieć wiele profili (partner, dziecko, mama → analiza dla nich)
- `charts` z cache - generacja efemeryd jest deterministyczna, nie ma sensu re-compute. Wersjonujemy ephemeris_version.
- `readings.ai_prompt_version` - krytyczne. Jak zmieniasz prompt, chcesz wiedzieć kto dostał starą vs nową wersję, retroaktywnie regenerować dla wybranych userów.

## 6. Tech stack & integracje

### Stack (cherry-pick z Alcheme.io)
- **Frontend:** React 18 + Vite 5 + Tailwind + shadcn/ui. PWA z service worker (push notifications).
- **Backend:** Supabase (Postgres + Auth + Edge Functions + Storage).
- **AI:** Anthropic (Claude Sonnet 4.6 primary) + OpenAI (GPT-4o jako fallback). Wszystko przez serverside edge function, never expose API keys.
- **Astrology computation:** Swiss Ephemeris (gold standard, używają wszyscy: Astro.com, Co-Star). Library: `swisseph-wasm` lub Python `pyswisseph` w serverless function. **Decyzja:** sprawdzić co już używasz lokalnie, prawdopodobnie zostaje.
- **Geocoding:** Google Places API (autocomplete + lat/lon + timezone). Free tier do 200 reqs/mc, potem $17/1000 reqs. Cache geocoded miasta agresywnie.
- **Stripe:** Subscription + Checkout. Tax handling via Stripe Tax (PL VAT 23%).
- **Analytics:** PostHog (free tier do 1M events/mc). Funnel + cohorts + replays.
- **Email:** Resend (transactional) + ConvertKit / Loops dla newsletter.
- **Push notifications:** Web Push API (PWA). Dla iOS - trzeba PWA installed (iOS 16.4+ supports).
- **Hosting:** Vercel (frontend) + Supabase (backend). Łącznie ~$50-100/mc do 1000 userów.

### Koszty AI (estymata MVP, 100 paid users)
| Use case | Volume / mc | Avg tokens (in+out) | Model | Koszt |
|---|---|---|---|---|
| Natal interpretation | 100 (raz na user) | 2000+1500 | Sonnet 4.6 | ~$1.50 |
| Daily reading | 100 × 30 = 3000 | 1000+800 | Sonnet 4.6 | ~$30 |
| Synastry | 100 × 2 ~ 200 | 3000+2000 | Sonnet 4.6 | ~$5 |
| **TOTAL** | | | | **~$37/mc** |

Z 100 paid x 29 zł = 2900 zł = ~$725 revenue. AI = ~5% kosztu. **OK, skaluje.**

Trick na koszty: **cache daily transits per chart** - tranzyty dnia są deterministyczne, generuj interpretację raz, serwuj wielu userom z podobnym chartem. Trudne (każdy chart unikalny) ale można generować "tranzyt-base" raz na dzień (Mars vs Sun, Mars vs Moon... 100 takich) i potem dla usera tylko składać które go dotyczą. Tylko zaawansowane - na P1, nie MVP.

## 7. AI prompty - specyfikacja (krótko, szczegóły w osobnym pliku)

Prompty są kluczowym IP produktu. Każdy prompt jest:
- **Wersjonowany** (`natal-v1.0`, `natal-v1.1`...)
- **Testowany** golden set'em (min 10 par input/output)
- **Reviewed przez koleżankę** przed prod release

Zobacz: `prompts-v1.md` (osobny plik) dla pełnych promptów.

**Architektura promptów (skróty):**
1. **System prompt** (zdefiniowany przez koleżankę - jej głos, jej zasady) - ten sam dla wszystkich call'i
2. **User prompt template** per use case (natal / daily / synastry) - z placeholders na dane
3. **Output format** - structured (markdown z określoną strukturą, łatwe do parsowania w UI)

**Edge cases promptów:**
- Bez godziny urodzenia → prompt dostaje flagę `birth_time_unknown=true` i instrukcję "nie wspominaj domów ani znaku wschodzącego, focus na planetach w znakach"
- Conflict aspectów (np. 10 aspectów napiętych) → instrukcja "wybierz top 3 najsilniejsze, nie wymieniaj wszystkich"
- Sensitive życiowe (śmierć, choroba, separacja w kontekście) → safety guardrail "nie diagnozuj, nie przepowiadaj, oferuj refleksję"

## 8. Monetyzacja - rekomendacja

### Model: Pure subscription, jeden tier (na MVP)

**Tier "Cosmogram Plus": 29 zł/mc lub 290 zł/rok (~24 zł/mc, -17%)**

Co zawiera:
- Pełna interpretacja natalna (vs. teaser darmowy)
- Daily reading (unlimited)
- Astro-Match (do 5/mc, potem +5 zł / match albo unlimited w wyższym tierze później)
- Year-ahead overview (raz na rok przy subskrypcji)

Free tier:
- Natal chart (wizualizacja)
- Teaser interpretacji (Sun/Moon/Rising, ~300 słów)
- 1 daily reading (sample, żeby zobaczyć co dostają)
- 1 Astro-Match w trialu

**Trial:** 7 dni darmowo, karta wymagana (lower-friction signup variant: bez karty, ale niższy convert). A/B test.

### Dlaczego TYLKO subscription, nie hybrid
1. **Habit formation > transaction.** Daily reading = daily open = retention. Sub model wzmacnia habit.
2. **Predykcja revenue** = łatwiejsze planowanie AI kosztów.
3. **One-time pay** psychologicznie konkuruje z sub w głowie usera (dlaczego mam płacić 29/mc skoro mogę 15 zł raz?).
4. **Co-Star, Pattern, Sanctuary** - wszystkie subscription pure. Działa.

### Deep dive PDFs - w Phase 2 jak BONUS, nie konkurencja
W P2, gdy są już subs, dodaj **one-time premium reports** (39-79 zł) dla:
- Year ahead (rozbudowany, 30-stronicowy)
- Złoty Kompas (career + finance, 20 stron)
- Bagaż Karmiczny

Tylko dla subskrybentów (lub z 30% rabatem dla nich vs non-subs). To boost ARPU, nie alternatywa do sub.

## 10. Plan testów

### 10.1 Testy techniczne (standard)
- **Unit tests** dla astro computation (porównaj z Astro.com dla 10 znanych dat - musi się zgadzać do 0.1°)
- **Integration tests** dla flow: signup → onboarding → first reading → save
- **E2E (Playwright)** dla critical paths: signup → trial → reading, signup → match → paywall
- **Load test** prompt edge function: 100 concurrent requests, czy edge function skaluje

### 10.2 Prompt evaluation (krytyczne, AI jest produktem)
**Golden set** - 10-15 par (input chart, expected interpretation themes). Robić z koleżanką w W3.

Każda zmiana prompta → przepuszczamy przez golden set → koleżanka rates na 1-5 (autentyczność, accuracy, voice). 

**Acceptance criteria nowego prompta:**
- Średnia ocena ≥ 4.0/5 od koleżanki
- Zero "hallucinated technical claims" (np. wymyślona pozycja planety)
- Zero red-flagged content (medical/legal/financial advice w niewłaściwym tonie)

### 10.3 Astrolog quality review (per sprint)
Po każdym sprincie - koleżanka czyta 5-10 fresh outputs (real users, anonimizowane), rates blind. Trend w czasie = sygnał czy prompt się degraduje albo poprawia.

### 10.4 User panel testing
- W5: 5 friends/family - feedback usability + tone
- W8: 20 waitlistowych - free MVP access, 1-week feedback
- W12: launch z 50 paid testers, NPS po 7 dniach

### 10.5 A/B tests (rolling, od W13)

## 11. Edge cases & risks

### Funkcjonalne edge cases
- **Brak godziny urodzenia** → solar chart fallback (sekcja 4, F2)
- **Niepewna godzina (only "rano/popołudniu")** → user wpisuje 9:00 / 15:00 / 21:00, oznaczone jako "przybliżona" w UI
- **Miejsce urodzenia niejednoznaczne** (Warszawa MO/Warszawa USA) → autocomplete z weryfikacją lat/lon
- **Historyczne daty z DST issues** → Swiss Ephemeris handluje, ale spot-check 1940-1980 dat (Polska zmieniała DST chaotycznie)
- **Krańcowe szerokości geograficzne** → houses calculation fails powyżej koła polarnego. Default na Whole Sign w takich przypadkach.
- **Future dates** (kiedy user wpisze złą datę) → walidacja: max = dzień dzisiejszy
- **Stała data urodzenia + zmiana ephemeris version** → re-compute trigger, oznaczyć w UI że "zaktualizowaliśmy precyzję"

| iOS PWA push not working | Med | Med | Ostrzeżenie w UI, push tylko Android/desktop na MVP |
